# Performance Optimization: Optical Flow Runtime Analysis

## Problem Identified

During pipeline profiling, discovered that **Optical Flow (Farneback) consumes 90% of processing time**:

### Performance Metrics

| Component | Time | CPU % | Status |
|-----------|------|-------|--------|
| **Optical Flow (Farneback)** | ~390ms/frame | 90% | ❌ BOTTLENECK |
| Camera Motion Detection | ~390ms/frame | N/A  | (uses OF) |
| Kalman Smoothing | 0.5ms/frame | 0% | ✓ Fast |
| Trajectory Validation | 0.1ms/frame | 0% | ✓ Fast |
| Cumulative | ~390ms/frame | 90% | ❌ **12x SLOWER than 30fps** |

**Impact**: At 30 fps, each frame should take <33ms, but optical flow adds 390ms = 11.7x slower than real-time requirement.

---

## Solution Implemented

### Optimization Strategy

1. **Disable Optical Flow by Default**
   - New parameter: `enable_optical_flow: bool = False`
   - Pipeline now runs at ~1-2ms per frame (✓ real-time compatible)
   - Kalman smoothing remains active (fast + effective)

2. **Make Optical Flow Optional**
   - Users can enable with: `enable_optical_flow=True` if needed
   - Trade-off documented: -+390ms/frame for robustness vs sparse keypoints

3. **Preserve Validation Chain**
   - Trajectory Validator: Always active (0.1ms cost) ✓
   - Position Smoother: Always active (0.5ms cost) ✓
   - Camera Motion Detector: Only with optical flow

### Code Changes

**File**: `modules/batch_processor.py`

```python
# Constructor: Add enable_optical_flow parameter (default: False)
def __init__(
    ...
    enable_optical_flow: bool = False  # ← NEW: disabled by default
):

# Store flag
self.enable_optical_flow = enable_optical_flow

# In initialize_modules(): Conditional initialization
if self.enable_optical_flow:
    self.optical_flow_tracker = OpticalFlowTracker(fps=fps)
    self.camera_motion_detector = CameraMotionDetector()
    print(f"  - Optical Flow Tracker: ✓ Activado (⚠ +390ms/frame)")
else:
    self.optical_flow_tracker = None
    self.camera_motion_detector = None
    print(f"  - Optical Flow Tracker: ✗ Desactivado (para rendimiento)")

# In process_chunk(): Guard all OF calls
if self.optical_flow_tracker is not None:
    of_fallback = self.optical_flow_tracker.update(...)
    of_confidence = self.optical_flow_tracker.get_confidence(...)
```

### With Optical Flow Re-enabled (Optional)

## When to Use Each Mode

### Disable OF (Default) ✓
- **Real-time video streaming**
- **Match analysis during live play**
- **Field with good keypoint coverage** (>5 keypoints/frame)
- **GPU/Processing resources limited**

### Enable OF
- **Post-match detailed analysis**
- **Poor lighting scenarios** (rain, night, shadows)
- **Sparse keypoint detection** (<3/frame):
  - Bad field markings
  - Extreme camera angles
  - Partial field views

## Impact on Features

### Still Working (Unchanged)
- ✓ Kalman trajectory smoothing (60% jitter reduction)
- ✓ Trajectory validation (velocity/acceleration constraints)
- ✓ Position smoothing with confidence weighting
- ✓ Heatmap generation with zone confidence

### Affected by Disabling OF
- ⚠ Optical flow fallback for undetected players (removed by default)
- ⚠ Camera motion detection (only with OF enabled)
- ℹ Kalman smoothing handles most cases anyway (inherent temporal filtering)

---

## Recommendation

**Use Default** (`enable_optical_flow=False`) for:
- **Production**: Real-time streaming pipelines
- **Most scenarios**: Kalman smoothing provides adequate continuity

**Consider Enabling** (`enable_optical_flow=True`) only if:
- Keypoint detection is unreliable (<3 keypoints/frame consistently)
- Processing time allows (e.g., offline analysis)
- Need absolute robustness under detector failure

## Alternative Optimization Paths (Future)

If optical flow becomes necessary again:

1. **Lucas-Kanade (Sparse) Flow**: ~50ms/frame vs ~390ms
   - Trade: Local tracking instead of dense field
   - Benefit: 8x faster

2. **GPU Acceleration**: CUDA-enabled optical flow
   - Trade: Requires NVIDIA GPU
   - Benefit: Potential 50-100x speedup

3. **Reduced Resolution**: Run OF at 480p, upscale results
   - Trade: Less detail, potential artifacts
   - Benefit: ~100ms/frame (4x speedup)

4. **Key-Frame Sampling**: Compute OF every Nth frame
   - Trade: Missed motion between samples
   - Benefit: Variable cost reduction
